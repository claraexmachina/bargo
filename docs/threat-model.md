# Bargo V3 — Threat Model

## Trust model: what changed from V2

V2 described an operator that received reservation prices as plaintext, held them in a database for approximately 15 seconds, and purged them on deal completion. The operator was a trusted-but-auditable broker.

V3 removes plaintext from the service entirely. Clients encrypt reservation prices and natural-language conditions with X25519 ECDH before sending them over the wire. The service stores only the AEAD ciphertext envelopes (`enc_*` columns in SQLite — no `ask_price`, no `bid_price`, no `plaintext_*` columns anywhere). Plaintext is reconstructed inside the service **only in ephemeral request-scope memory** — the `decryptReservationEphemeral` call — for the duration of packaging the NEAR AI prompt. It is discarded immediately after. It is never written to disk, never written to the database, and never printed to logs.

The NEAR AI TEE trust anchor remains unchanged: Intel TDX + NVIDIA GPU attestation on every inference, independently verifiable by any judge.

## Threat table

| # | Attacker | Capability | Defense | Residual risk |
|---|---|---|---|---|
| 1 | Malicious NEAR AI operator | Swap model mid-inference; craft arbitrary JSON response | `signed_response` binds `model`, `nonce`, and `completion_id`. Verifier re-checks TDX measurement (MR_TD pin) + NVIDIA NRAS GPU evidence against pinned values. Nonce = `keccak256(dealId ‖ completionId)` is unique per inference. | Compromise of Intel PCS or NVIDIA root of trust (hardware RoT — implausible but outside our control) |
| 2 | Malicious Bargo service operator | Read in-flight plaintext during negotiation | Plaintext **never persists** to DB, logs, or filesystem. The only exposure window is the ~10 ms ephemeral decrypt inside `decryptReservationEphemeral`. An operator who modifies source code to extend or log this window also invalidates the open-source audit trail, making the modification detectable. | Source-code trust: if the operator deploys modified binaries without publishing them, the open-source audit trail is broken. Mitigated by public repo; reproducible builds are planned post-hackathon. |
| 3 | Counterparty (buyer or seller snooping on the other side) | Learn the other party's reservation price or raw condition text | Service returns only public fields. `agreedPrice` is the only price ever revealed (emitted on-chain at settlement). `AgreedConditions` (the merged meetup result) is shown to both parties, never the other side's raw condition text. No API endpoint returns `encMinSell`, `encMaxBuy`, or any decrypted condition string. | User inadvertently embedding reservation price in condition text (e.g., "I will accept no less than 700,000 KRW") — mitigated by UI placeholder copy and a client-side length warning, but unenforceable server-side. |
| 4 | Stale attestation replay | Replay a valid `nearAiAttestationHash` from deal X onto deal Y | `nonce = keccak256(dealId ‖ completionId)` is embedded in the attestation bundle and re-derived by the off-chain verifier at step 2. `dealId = keccak256(listingId ‖ offerId)` is collision-resistant. An attestation for deal X can only satisfy deal X. | None — nonce binding is cryptographically sound given SHA-3 collision resistance. |
| 5 | DB snapshot theft | Exfiltrate the SQLite file (at rest or via backup) | All reservation data is stored as AEAD-protected `enc_*` blobs (X25519 ECDH + HKDF-SHA256 + XChaCha20-Poly1305). No plaintext columns exist anywhere in the schema. Completed and active deals are equally opaque to a DB-only attacker. No master-key backdoor. | Theft of the service's X25519 private key (`SERVICE_DECRYPT_SK`) in combination with the DB snapshot allows decryption of all `enc_*` blobs. Key rotation per ops policy limits the exposure window. |
| 6 | Service crash or restart during negotiation | Negotiation stuck in `running` state; plaintext re-derived on restart | Bootstrap recovery sets any negotiation stuck in `queued` or `running` for more than 120 seconds to `fail`. No plaintext is ever written to disk during the crash window, so restart cannot leak plaintext across the crash boundary. | Buyer must resubmit offer after a crash. No silent state corruption. |
| 7 | Relayer key leak | Call `settleNegotiation` with arbitrary `agreedPrice` and `nearAiAttestationHash` | Forged settlements are detectable via the off-chain verifier: the forged `nearAiAttestationHash` will fail NEAR AI attestation verification (no real inference occurred). A buyer who locks escrow on a forged settlement is the only direct victim; the forgery is discoverable before lockEscrow via the verifier script. Emergency relayer rotation via `setAttestationRelayer` (owner-only). | Attacker can create fake settled deals on-chain. Fakes have no backing NEAR AI attestation so the verifier flags them. |
| 8 | RLN bypass (spam offers) | Submit many offers to probe seller's floor price via repeated matches | Nullifier deduplicated per epoch in the `rln_nullifiers` table. On-chain `RLNVerifier.verify()` rejects duplicate nullifiers. Karma tier gate requires SNT stake for high-value listings. Note: the current V3 stub accepts any structurally valid proof; full Status Network RLN SDK integration (real ZK verification) is Phase 2 work. | Tier 0 users can create multiple wallets, but Karma tier gating imposes economic cost at scale. Real ZK RLN verification planned for production. |
| 9 | Intent conditions leak (matchmaker) | Learn a buyer's standing intent conditions via the matchmaker path | The matchmaker follows the same ephemeral decrypt pattern as the main negotiation flow. `decryptIntentConditions` reconstructs buyer conditions in local memory, passes them to NEAR AI, then explicitly zeroes the variable (`buyerConditions = ''`) before returning. Conditions are never logged (only `intentId`, `listingId`, and `score` are logged). `matchReason` (the NEAR AI explanation returned to the buyer) is a short public-facing string that does not contain raw condition text. | Same source-code trust residual as Row 2: a modified binary that extends the ephemeral window. |
| 10 | Malicious buyer probing seller floor via repeated bids | Submit offers with impossibly-high ceilings to learn seller's minimum via binary search | Even if the buyer wins every negotiation, only `agreedPrice` is revealed at settlement — not the seller's `encMinSell`. The ZOPA midpoint calculation uses both bounds, but the output (agreed price) does not algebraically reveal the seller floor given only a single data point. The buyer would need many data points with varying ceilings, but each offer requires a distinct listing (not possible to re-offer on the same listing) and consumes Karma throughput. | Theoretical multi-listing correlation attack at very high Karma throughput. Throughput limits make this prohibitively expensive at scale. |

## What we explicitly do NOT defend against

1. **Pre-settle DB access combined with service private key theft** — An attacker who simultaneously exfiltrates the SQLite DB and the service's `SERVICE_DECRYPT_SK` can decrypt all `enc_*` blobs. This is the primary residual risk in V3; it is honestly disclosed here and not elsewhere minimised.

2. **Malicious client-side RLN proof generation** — The V3 service uses a stub RLN verifier that accepts any structurally valid proof. Real ZK proof verification against the Status Network circuit is Phase 2. Until then, a sophisticated attacker can craft fake proofs that pass the stub check.

3. **User pasting reservation price into condition text** — If a user writes their floor price in free-text conditions (e.g., "I won't go below 700,000 KRW"), that plaintext is passed to NEAR AI inside the TEE. The TEE does not leak it to the counterparty, but it does flow through the ephemeral decrypt window, and the NEAR AI operator can observe it in the inference prompt. The UI warns against this; server-side enforcement is not possible without breaking the natural-language condition feature.

4. **NEAR AI's Intel and NVIDIA roots of trust** — The verifier trusts Intel PCS and NVIDIA NRAS as hardware roots of trust. A compromise of either would invalidate the attestation chain. This is an explicit assumption shared with the hackathon track requirements.

5. **Honest NEAR AI MR_TD publication** — The `MR_TD` pin must be sourced from NEAR AI's official documentation. If NEAR AI publishes an incorrect measurement, the pin provides no additional protection. The ECDSA signature and NRAS GPU evidence checks (Rows 1, 3) remain valid independently.
